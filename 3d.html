03-3. 크롤링·처리 파이프라인

수집 이후 run_once 기준 — 하드필터부터 캐시·DB·통계까지

핫딜연구소
상세

1. 하드 필터 _apply_hard_filter

  • 드롭 키워드 — 제목에 종료·품절·광고 등 → 즉시 DROP
  • 댓글 — 3 미만이면 원칙 DROP. 예외: 게시 30분 이내 + 댓글 ≥1 → 통과 (kept fresh deal 로그)
  • 가격 — 상세·제목에서 추출·정규화, 없으면 DROP
  • 통과 시 process_deal이 다음 단계로 진행

2. 소프트 스코어 + 네이버 _calculate_soft_score

  • 가격 명시, 할인·긍정 키워드, 댓글 수, 부정 키워드, 반응 속도(velocity) 등 가감점
  • 네이버 쇼핑 API — 제목 정리 후 검색, 첫 상품 lprice와 비교. 자격·설정 없으면 비교 생략
  • 단가(수량·배송비 반영) — 딜이 네이버보다 비싸면 DROP 후 점수 계산 중단
  • 싸면 savings·할인율·절약액에 따른 보너스, deal.naver_price 설정(LLM·DB용)

3. 점수 임계값

  • score < 0 → DROP + 사유
  • 그 외 → status = "READY" → run_once에서 [READY] 출력 후 분석 대열에 포함

4. LLM 분석 Analyzer.analyze_batch

  • 5개 단위 청크, 청크 사이 15초 대기(RPM 완화)
  • 5건인 이유 — 429/503·레이트·모델 부하를 피하고, 청크가 클수록 한 번 실패 시 손실 단위·JSON 잘림·파싱 실패 위험이 커짐(analyzer 쪽 주석·예외 메시지와 동일 취지).
  • 입·출력 크기 — 항목마다 제목·가격·네이버·절약액·링크·점수·댓글 미리보기(상위 5개) 등으로 프롬프트가 두껍고, 구조화 응답도 길어져 TPM·비용·지연에 민감 → 배치 상한을 5로 둠.
  • 위 필드를 시스템 프롬프트에 묶어 청크별로 전송, 응답은 딜 ID로 매핑
  • 기본 gemini-3-flash-preview, 실패 시 gemini-2.0-flash 등 폴백·재시도
  • 응답 JSON 스키마: HOT/MAYBE만 배열에 넣고 DROP은 생략 유도

5. 청크 결과를 딜에 반영

  • 응답에 ID 있음 → is_hotdeal, 카테고리(enum 매핑), ai_summary, sentiment_score, embed_text, status를 HOT/DROP으로 설정
  • 없음(모델이 DROP) → is_hotdeal=False, DROP 처리, 요약은 고정 문구 등
  • 청크 전체 JSON 파싱 등 예외 시, 아직 READY인 딜은 status = "ERROR"

6. 임베딩 _generate_embeddings

  • is_hotdeal이고 embed_text 있는 딜만 gemini-embedding-001 → deal.embedding
  • 실패 시 로그만(상위에서 처리)

7. 파이프라인 마무리 run_once

  • Processor.update_cache — cache.json에 당일 날짜와 함께 저장
  • db.save_deal — is_hotdeal일 때만 DB(필드는 database.py 정의)
  • 콘솔 최종 요약, last_run_result.json 전체 딜 덤프
  • db.save_app_stats — 커뮤니티 수, 수집·드롭·캐시·핫딜(신규+캐시)·절약액 합 등
  • db.clean_old_deals(5) — 5일 지난 딜 정리
한 줄 흐름: 하드필터 → 소프트 스코어+네이버 → READY 선별 → 청크 LLM → 매핑·암시적 DROP → 핫딜 임베딩 → 로컬 캐시·DB(핫만)·JSON·통계·오래된 행 삭제
