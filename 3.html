팩트: Playwright로 게시판·상세 페이지를 읽는 수집 구간에는 Gemini 호출이 없습니다. 아래 모델들은 수집이 끝난 뒤(스크래퍼 배치) 또는 웹 앱 서버(LAB)에서만 호출됩니다.
HOTDEAL_SCRAPER
gemini-3-flash

배치 LLM — 딜 단위 판별·요약

  • 위치: app/core/analyzer.py · Analyzer.analyze_batch (generateContent)
  • 시점: run_once.py 흐름에서 수집 + Processor(하드/소프트·READY) 이후. READY 상태 딜만 청크로 묶어 전송합니다.
  • 입력: 제목·가격·네이버 비교·절약액·점수·댓글 미리보기 등을 시스템 프롬프트에 포함.
  • 출력: 구조화 JSON — 딜별 HOT/MAYBE만 배열에 넣고 DROP은 생략 유도. 응답에서 카테고리, ai_summary, sentiment_score, 검색용 embed_text 등을 채웁니다.
  • 운영: 기본 5건 청크·청크 사이 15초(RPM 완화). 실패 시 gemini-2.0-flash로 폴백·429/503 재시도 로직.
HOTDEAL_SCRAPERLAB
gemini-embedding-001

벡터 생성 — DB 적재 & 추천 질의

  • SCRAPER: 같은 Analyzer 경로의 _generate_embeddings — is_hotdeal이고 embed_text가 있는 딜만 embedContent → 768차원을 deal.embedding에 넣고 Supabase hotdeals upsert 시 함께 저장.
  • LAB: src/app/actions.ts의 getRecommendationFromPrompt에서 사용자 문장(프롬프트)을 taskType: RETRIEVAL_QUERY, outputDimensionality: 768로 임베딩 → supabase.rpc('match_hotdeals', …)의 query_embedding으로 넘김.
HOTDEAL_LAB
gemini-2.5-flash

추천 검색 — 질의를 구조화

  • 위치: actions.ts · getRecommendationFromPrompt 첫 단계 (generateContent).
  • 용도: 자연어 한 줄을 시스템 지시에 맞춰 JSON으로 풀기 — 예: target_categories, search_keywords, related_keywords, 공감 한 줄 ai_comment. 이후 같은 문자열을 위 임베딩 단계에 넘겨 벡터 검색과 연결합니다.
  • 보조: 할당량·429 등에서 gemini-2.5-flash-lite로 폴백하는 분기가 코드에 있습니다.

일반 목록·베스트 조회는 Supabase 조회만 하며, 이 모델은 추천 검색 UI를 탈 때만 씁니다.
