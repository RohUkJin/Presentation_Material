05-1. 코드 기준 종단간 흐름

HOTDEAL_SCRAPER · HOTDEAL_LAB 저장소 기준 — 수집부터 추천 검색까지 다섯 축

핫딜연구소
E2E

1 크롤링 방식

  • Playwright chromium.launch(headless=True) + 단일 BrowserContext에서 소스별 크롤러 실행 (run_once.py)
  • 소스: Community1_Crawler · Community2_Crawler · Community3_Crawler — 각각 process(context)
  • BaseCrawler: 게시판 goto(..., wait_until='domcontentloaded') → 목록 crawl_list(페이지네이션) → 행마다 상세 crawl_detail
  • 상세 HTML은 커뮤니티별로 BeautifulSoup 등으로 필드 추출
  • Processor._apply_hard_filter에 걸린 행은 상세 크롤 생략으로 부하 절감

2 AI가 들어가는 지점

  • 페이지 수집 단계에는 LLM 없음 — 브라우저로 글·가격·댓글 메타만 구조화
  • 수집 후 Processor.process_deal(규칙·네이버 단가·스코어) → READY만 Analyzer.analyze_batch로 Gemini 판별·요약·감성·embed_text
  • 핫딜로 확정된 항목에 gemini-embedding-001 → deal.embedding (768차원, DB·검색용)
정리: AI는 “크롤 루프 안”이 아니라 배치 파이프라인의 분석·임베딩 구간에 위치합니다.

3 DB 적재 과정

  • Python Supabase 클라이언트 (app/core/database.py) — hotdeals 테이블에 upsert(id 기준)
  • run_once 루프에서 deal.is_hotdeal일 때만 db.save_deal(deal) 호출
  • 페이로드: title·link·embedding·ai_summary·sentiment_score 등 스키마에 맞게 매핑
  • 실행 요약은 crawl_stats에 insert, 오래된 행은 clean_old_deals(5)로 정리

4 클라이언트로 데이터 가져오기

  • 웹은 Next.js 16 — UI에서 서버 함수(Server Actions) (src/app/actions.ts) 호출
  • 목록·베스트10·날짜별 인기 등은 서버에서 supabase.from('hotdeals').select(...)로 조회 후 필터·정렬·중복 제거하여 반환
  • 브라우저에 DB 키를 직접 심는 패턴이 아니라, 서버 측 Supabase 클라이언트로 조회하는 형태(기존 05 슬라이드와 동일 선상)

5 클라이언트 추천 검색 원리

  • 컴포넌트가 getRecommendationFromPrompt(prompt) Server Action 호출 (RecommendationPopup.tsx)
  • 1) gemini-2.5-flash로 자연어 질의를 JSON으로 구조화 — 카테고리·검색 키워드·연관 키워드·한 줄 코멘트
  • 2) 같은 질의 문자열을 gemini-embedding-001에 taskType: RETRIEVAL_QUERY, 768차원으로 임베딩
  • 3) supabase.rpc('match_hotdeals', { query_embedding, match_threshold, match_count, filter_categories }) — pgvector 코사인 유사도로 후보 추출
  • 4) 커뮤니티 링크 제거 등 후처리 후 상위 20건 반환 · RPC가 이미 유사도 순이라 순서 유지
